# 酒店管理系统前后端数据库对接规范

版本：v0.1  
适用范围：酒店管理系统前端、Python 后端、MySQL 数据库联调  
依据文档：《系统需求分析》《概念&逻辑结构设计》《物理结构设计&系统实现》

## 1. 项目边界

本系统面向中小型酒店日常经营管理，核心业务包括客户管理、客房管理、预定管理、入住管理、换房、结账收银、清扫任务、账号管理和操作审计。

系统角色：

| 角色 | 说明 | 主要权限 |
| --- | --- | --- |
| 管理员 | 系统管理人员 | 账号审核、账号管理、客房管理、客户管理、操作审计 |
| 前台工作人员 | 前台业务人员 | 客户录入、预定、入住、换房、订单查询、结账 |
| 保洁员 | 清扫任务执行人员 | 查看清扫任务、开始清扫、完成清扫 |
| 审计员 | 审计查询人员 | 操作日志查询、账单查询 |
| 顾客 | 酒店服务对象 | 不作为后台登录用户 |

## 2. 总体架构

```text
前端 Web/桌面界面
    |
    | HTTPS + JSON REST API
    v
Python 后端服务
    |
    | SQLAlchemy/PyMySQL + 事务 + 存储过程/视图
    v
MySQL 8.0 / InnoDB / HotelSys
```

技术默认约定：

| 层 | 默认技术 | 说明 |
| --- | --- | --- |
| 前端 | 任意 Web 框架 | 只调用后端 REST API，不直接连接 MySQL |
| 后端 | Python + FastAPI | 若团队已选 Flask/Django，可沿用本文接口语义 |
| 数据库 | MySQL 8.0 + InnoDB | 数据库名 `HotelSys`，字符集 `utf8mb4` |
| 鉴权 | JWT Bearer Token | 登录成功后由后端签发 |
| 密码 | bcrypt 或 argon2 哈希 | 不允许明文入库 |
| 金额 | 后端 `Decimal`，API 字符串 | 避免前端浮点精度问题 |
| 时间 | ISO 8601 | API 使用 `YYYY-MM-DDTHH:mm:ss+08:00`，落库为 MySQL `DATETIME` |

职责边界：

| 对接方 | 责任 |
| --- | --- |
| 前端 | 页面展示、表单校验、调用 API、展示后端错误信息 |
| 后端 | 登录鉴权、权限控制、业务校验、事务控制、调用数据库视图/表/存储过程 |
| 数据库 | 表结构、约束、索引、视图、存储过程、触发器、基础字典数据 |

前端不得直接读写数据库。所有新增、修改、删除类操作必须经由后端 API，以便统一做权限控制、操作日志和事务处理。

## 3. 命名与数据格式

### 3.1 命名风格

| 场景 | 风格 | 示例 |
| --- | --- | --- |
| 数据库表名 | PascalCase | `Customer`, `CheckIn`, `CleaningTask` |
| 数据库字段名 | PascalCase | `CustomerId`, `ReserveStartTime` |
| 后端 Python 变量 | snake_case | `customer_id`, `reserve_start_time` |
| HTTP JSON 字段 | lowerCamelCase | `customerId`, `reserveStartTime` |
| REST 路径 | kebab-case / 复数名词 | `/api/v1/room-types` |

### 3.2 通用响应格式

成功响应：

```json
{
  "code": 0,
  "message": "success",
  "data": {},
  "requestId": "req-20260608-000001"
}
```

分页响应：

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "items": [],
    "total": 0,
    "page": 1,
    "pageSize": 20
  },
  "requestId": "req-20260608-000002"
}
```

失败响应：

```json
{
  "code": 40901,
  "message": "客房当前不是空闲状态，不能办理入住",
  "data": null,
  "requestId": "req-20260608-000003"
}
```

### 3.3 通用错误码

| code | 含义 | HTTP 状态 |
| --- | --- | --- |
| 0 | 成功 | 200/201 |
| 40001 | 参数格式错误或必填缺失 | 400 |
| 40002 | 业务字段取值非法 | 400 |
| 40101 | 未登录或 token 无效 | 401 |
| 40301 | 当前角色无权限 | 403 |
| 40401 | 资源不存在 | 404 |
| 40901 | 业务状态冲突 | 409 |
| 40902 | 唯一约束冲突 | 409 |
| 50000 | 系统内部错误 | 500 |

### 3.4 分页、排序、过滤

查询接口统一支持：

| 参数 | 说明 | 默认 |
| --- | --- | --- |
| `page` | 页码，从 1 开始 | 1 |
| `pageSize` | 每页数量，最大 100 | 20 |
| `sort` | 排序字段 | 后端决定 |
| `order` | `asc` 或 `desc` | `desc` |

时间范围统一使用 `startTime`、`endTime`。身份证号使用 `cardId`，手机号使用 `phone`。

## 4. 状态枚举

前端必须使用以下状态值，不得自行增加自然语言状态。

| 枚举 | API 字段 | 可选值 |
| --- | --- | --- |
| 房间状态 | `roomStatus` | `空闲`、`已预定`、`已入住`、`待清扫`、`停用` |
| 预定状态 | `reservationStatus` | `未入住`、`已入住`、`已取消` |
| 入住订单状态 | `orderStatus` | `进行中`、`已结束`、`已取消` |
| 账号审批状态 | `approvalStatus` | `待审核`、`已启用`、`已驳回`、`已停用` |
| 清扫状态 | `cleanStatus` | `待清扫`、`清扫中`、`已完成` |
| 角色名称 | `permissionName` | `管理员`、`前台工作人员`、`保洁员`、`审计员` |

房态核心流转：

```text
空闲 -> 已入住 -> 待清扫 -> 空闲
```

说明：当前设计中，预定阶段主要锁定客房类型，不直接绑定具体房间，因此通常不改变具体房间状态为 `已预定`。`已预定` 作为扩展状态保留。

## 5. 数据库对接约定

### 5.1 数据库对象

数据库名：`HotelSys`

核心表：

| 表名 | 中文含义 | 主要用途 |
| --- | --- | --- |
| `Customer` | 客户表 | 客户基础资料、身份证号、折扣 |
| `Discount` | 折扣表 | 折扣等级、折扣率 |
| `AccountPermission` | 账号权限表 | 角色和权限范围 |
| `UserAccount` | 用户账号表 | 后台登录账号、密码哈希、审批状态 |
| `RoomType` | 客房类型表 | 房型名称和价格 |
| `Room` | 客房信息表 | 房间号、楼层、电话、房型、房态 |
| `Reservation` | 预定信息表 | 预定订单 |
| `CheckIn` | 入住信息表 | 入住订单 |
| `Checkout` | 结账记录表 | 结账账单快照 |
| `OperationType` | 操作类型表 | 日志类型字典 |
| `OperationLog` | 操作记录表 | 关键操作审计日志 |
| `CleaningTask` | 清扫任务表 | 退房后清扫任务 |

高频查询视图：

| 视图 | 用途 |
| --- | --- |
| `v_available_room` | 查询空闲客房 |
| `v_room_status_overview` | 查询房态统计 |
| `v_reservation_detail` | 查询预定详情 |
| `v_checkin_detail` | 查询入住详情 |
| `v_checkout_bill` | 查询结账账单 |
| `v_cleaning_task_queue` | 查询清扫任务队列 |
| `v_operation_audit` | 查询操作审计 |

业务存储过程：

| 过程 | 后端调用场景 |
| --- | --- |
| `sp_add_operation_log` | 后端内部写操作日志 |
| `sp_create_reservation` | 创建预定订单 |
| `sp_check_in` | 办理入住或预定转入住 |
| `sp_checkout` | 办理结账 |
| `sp_start_cleaning` | 开始清扫 |
| `sp_finish_cleaning` | 完成清扫 |
| `sp_operation_audit` | 操作审计查询 |

触发器：

| 触发器 | 作用 |
| --- | --- |
| `trg_checkin_before_insert` | 入住前校验客房必须为空闲 |
| `trg_checkin_after_insert` | 入住后更新房态、预定状态 |
| `trg_checkout_after_insert` | 结账后结束订单、生成清扫任务、置房态为待清扫 |
| `trg_cleaning_after_update` | 清扫完成后释放房间 |
| `trg_account_status_after_update` | 账号审批状态变化写审计日志 |

### 5.3 事务要求

以下操作必须在后端开启数据库事务：

| 操作 | 事务内容 |
| --- | --- |
| 创建预定 | 写 `Reservation`，写 `OperationLog` |
| 办理入住 | 校验房态，写 `CheckIn`，更新 `Room`，可能更新 `Reservation`，写日志 |
| 换房 | 锁定入住订单和新旧房间，更新 `CheckIn.RoomId`，更新新旧房态，写日志 |
| 结账 | 写 `Checkout`，触发结束入住订单、生成清扫任务、更新房态，写日志 |
| 开始清扫 | 锁定清扫任务，更新状态和保洁员，写日志 |
| 完成清扫 | 锁定清扫任务，更新状态和完成时间，触发释放房间，写日志 |
| 账号审核 | 更新账号状态，写日志 |

客房分配、换房、结账、清扫任务处理必须使用行级锁或存储过程，避免并发重复分配房间。

## 6. 鉴权与权限

### 6.1 登录

`POST /api/v1/auth/login`

请求：

```json
{
  "accountName": "front01",
  "password": "plain-password"
}
```

响应：

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "accessToken": "jwt-token",
    "tokenType": "Bearer",
    "expiresIn": 7200,
    "account": {
      "accountId": 1,
      "accountName": "front01",
      "permissionName": "前台工作人员",
      "approvalStatus": "已启用"
    }
  },
  "requestId": "req-20260608-000004"
}
```

登录规则：

| 条件 | 处理 |
| --- | --- |
| 账号不存在 | 返回 `40101` |
| 密码错误 | 返回 `40101` |
| 账号不是 `已启用` | 返回 `40301` |
| 登录成功 | 签发 JWT |

### 6.2 请求头

除登录、注册外，所有接口必须携带：

```http
Authorization: Bearer <accessToken>
Content-Type: application/json
```

## 7. REST API 规范

接口基准路径：`/api/v1`

### 7.1 账号与权限

| 方法 | 路径 | 角色 | 说明 |
| --- | --- | --- | --- |
| `POST` | `/accounts/register` | 未登录 | 注册账号，初始状态为 `待审核` |
| `GET` | `/accounts` | 管理员 | 查询账号列表 |
| `GET` | `/accounts/{accountId}` | 管理员 | 查询账号详情 |
| `PATCH` | `/accounts/{accountId}` | 管理员 | 修改账号基础信息 |
| `PATCH` | `/accounts/{accountId}/approval` | 管理员 | 审核、驳回、停用账号 |
| `GET` | `/permissions` | 管理员 | 查询角色权限 |

注册请求：

```json
{
  "accountName": "cleaner01",
  "password": "plain-password",
  "sex": "女",
  "birthday": "2000-01-01",
  "phone": "13800000000",
  "permissionId": 3,
  "supervisorId": 1
}
```

账号审核请求：

```json
{
  "approvalStatus": "已启用",
  "reviewComment": "资料完整，允许启用"
}
```

后端要求：

| 操作 | 要求 |
| --- | --- |
| 注册 | 校验 `AccountName` 唯一，密码哈希后写入 `UserAccount` |
| 审核 | 只允许状态写为 `已启用`、`已驳回`、`已停用` |
| 登录 | 只允许 `ApprovalStatus = 已启用` 的账号登录 |

| `GET` | `/customers` | 管理员、前台、审计员 | 按身份证、姓名、电话查询客户 |
| `GET` | `/customers/{customerId}` | 管理员、前台、审计员 | 查询客户详情 |

| `GET` | `/customers/{customerId}/history` | 管理员、前台、审计员 | 查询客户预定、入住、结账历史 |

```json
{
  "customerName": "张三",
```

### 7.4 预定管理

| 方法 | 路径 | 角色 | 说明 |
| --- | --- | --- | --- |
| `POST` | `/reservations` | 管理员、前台 | 创建预定 |
| `GET` | `/reservations` | 管理员、前台、审计员 | 查询预定 |
| `GET` | `/reservations/{reservationId}` | 管理员、前台、审计员 | 查询预定详情 |
| `PATCH` | `/reservations/{reservationId}/cancel` | 管理员、前台 | 取消未入住预定 |

创建预定请求：

```json
{
  "customerId": 1001,
  "roomTypeId": 2,
  "reserveStartTime": "2026-06-09T14:00:00+08:00",
  "reserveEndTime": "2026-06-11T12:00:00+08:00",
  "guestCount": 2,
  "prepayAmount": "200.00"
}
```

创建预定响应：

```json
{
  "reservationId": 3001,
  "reservationStatus": "未入住"
}
```

后端要求：

| 规则 | 说明 |
| --- | --- |
| 客户必须存在 | `customerId` 引用 `Customer` |
| 房型必须存在 | `roomTypeId` 引用 `RoomType` |
| 时间合法 | `reserveStartTime < reserveEndTime` |
| 入住人数 | `guestCount > 0` |
| 预付款 | `prepayAmount >= 0` |
| 默认状态 | `未入住` |
| 数据库调用 | 推荐调用 `sp_create_reservation` |

查询预定过滤参数：

| 参数 | 说明 |
| --- | --- |
| `customerId` | 客户 ID |
| `cardId` | 身份证号 |
| `reservationStatus` | 预定状态 |
| `roomTypeId` | 房型 ID |
| `startTime`、`endTime` | 预定入住时间范围 |

```json
  "customerId": 1001,
  "roomId": 2001,
  "reservationId": 3001,

  "checkInEndTime": null,
```

后端要求：

| 规则 | 说明 |
| --- | --- |
| 客户必须存在 | 引用 `Customer` |
| 房间必须空闲 | `Room.RoomStatus = 空闲` |
| 预定转入住 | `reservationId` 非空时，预定必须为 `未入住` |
| 默认订单状态 | `进行中` |
| 房态流转 | 新增入住后触发器置房间为 `已入住` |
| 数据库调用 | 推荐调用 `sp_check_in` |

换房请求：

```json
{
  "newRoomId": 2008,
  "oldRoomCleanRequired": true
}
```

换房规则：

| 规则 | 说明 |
| --- | --- |
| 入住订单 | 必须存在且 `orderStatus = 进行中` |
| 新房间 | 必须存在且 `roomStatus = 空闲` |
| 新房态 | 更新为 `已入住` |
| 原房态 | 默认更新为 `待清扫` 并生成清扫任务；若确认未使用可置为 `空闲` |
| 日志 | 写入 `办理入住` 或单独扩展 `换房` 操作类型 |

结账请求：

```json
{
  "checkoutTime": "2026-06-10T12:00:00+08:00"
}
```

结账响应：

```json

  "checkInId": 4001,
  "originalAmount": "560.00",
  "discountRateSnapshot": "0.9500",
  "discountAmount": "532.00",
  "prepayAmount": "200.00",
  "actualPayAmount": "332.00",
  "checkoutTime": "2026-06-10T12:00:00+08:00"
}
```

结账规则：

| 规则 | 说明 |
| --- | --- |
| 订单状态 | 只允许 `进行中` 订单结账 |
| 入住天数 | 最少按 1 天计算 |
| 原支付金额 | `房价 * 入住天数` |
| 折后应支付金额 | `原支付金额 * 折扣率` |
| 实际应支付金额 | `折后应支付金额 - 预支付金额`，可为负数，负数表示退款 |
| 房态流转 | 触发器将房态置为 `待清扫` |
| 清扫任务 | 触发器自动生成 `待清扫` 任务 |
| 数据库调用 | 推荐调用 `sp_checkout` |

## 8. 核心业务流程

### 8.1 预定流程

```text
前台查询客户
    |
客户不存在 -> 新增客户
    |
选择房型、预定时间、入住人数、预付款
    |
后端校验客户、房型、时间、金额
    |
创建 Reservation，状态为 未入住
    |
写操作日志 创建预定
```

### 8.2 预定转入住

```text
前台查询未入住预定
    |
按预定房型查询空闲客房
    |
选择具体房间
    |
后端校验房态为空闲
    |
创建 CheckIn，状态为 进行中，关联 ReservationId
    |
触发器更新 Room 为 已入住，Reservation 为 已入住
```

### 8.3 直接入住

```text
前台选择客户和空闲房间
    |
后端校验客户存在、房态为空闲、人数和预付款合法
    |
创建 CheckIn，ReservationId 为空
    |
触发器更新 Room 为 已入住
```

### 8.4 换房

```text
前台选择进行中的入住订单
    |
选择新空闲房间
    |
后端锁定入住订单、新房间、旧房间
    |
更新 CheckIn.RoomId 为新房间
    |
新房间置为 已入住
    |
旧房间默认置为 待清扫 并生成清扫任务
    |
写操作日志
```

### 8.5 结账和清扫

```text
前台选择进行中的入住订单
    |
后端计算账单金额
    |
创建 Checkout
    |
触发器更新 CheckIn 为 已结束
    |
触发器更新 Room 为 待清扫
    |
触发器创建 CleaningTask，状态为 待清扫
    |
保洁员开始清扫 -> 清扫中
    |
保洁员完成清扫 -> 已完成
    |
触发器更新 Room 为 空闲
```

### 9.2 后端分层

### 9.3 存储过程调用约定

后端优先调用数据库已有存储过程处理强一致业务：

| 后端服务方法 | 数据库存储过程 |
| --- | --- |

| `start_cleaning` | `sp_start_cleaning` |
| `finish_cleaning` | `sp_finish_cleaning` |

### 9.4 并发控制

| 场景 | 要求 |
| --- | --- |
| 办理入住 | 对目标房间行加锁，确认 `RoomStatus = 空闲` |

| 结账 | 对入住订单加锁，防止重复结账 |
| 清扫任务 | 对任务行加锁，防止重复开始或重复完成 |
