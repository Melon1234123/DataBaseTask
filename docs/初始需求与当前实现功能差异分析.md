# 初始需求与当前实现功能差异分析

## 1. 分析范围

本文对比以下两部分内容：

- 初始需求文档：`1 系统需求分析 改版.docx`，完成时间为 2026 年 5 月 9 日。
- 当前实现：仓库 `DataBaseTask` 的前端、FastAPI 后端和 `HotelSys` 数据库脚本，代码基线为提交 `d2c2e29`。

当前实现证据主要来自：

- 前端：`frontend/app.js`
- 后端路由：`backend/app/routers/`
- 后端业务层：`backend/app/services/`
- 后端数据访问层：`backend/app/repositories/hotel_repository.py`
- 数据库：`HotelSys/Schema.sql`、`Procedures.sql`、`triggers.sql`、`Views.sql`

“为什么发生变化”如果无法从需求或代码中直接确认，本文会标记为**工程推断**，不将其表述为项目组已经确认的决策。

## 2. 总体结论

初始需求中的七个核心业务模块均被保留：预定、入住、客房、账号、操作审计、客户和清扫任务，没有整块模块被完全删除。

22 项初始功能的实现情况如下：

| 状态 | 数量 | 说明 |
| --- | ---: | --- |
| 完整实现或等价合并 | 10 | 核心流程可以通过前端、后端和数据库共同完成 |
| 部分实现 | 12 | 后端已实现但前端缺少入口，或实现范围小于初始需求 |
| 完全未实现 | 0 | 没有整项 F01-F22 被彻底删除 |

实际变化可以概括为：

1. **核心闭环得到保留**：登录、客户建档、预定、入住、结账、自动生成清扫任务、清扫完成恢复房态可以运行。
2. **管理型功能有所收缩**：上级人员审批、账号资料编辑、客户资料编辑、折扣维护、房型/房间完整编辑删除等，在前端没有完整入口。
3. **查询功能前端化程度不足**：后端支持较丰富的筛选条件，但前端通常只提供状态或关键字筛选。
4. **预定库存校验未达到原需求**：当前没有按“房型 + 时间段”校验可预定数量，存在超卖风险。
5. **增加了课程演示和工程协作能力**：新增审计员、运营总览、预定取消、错误入住删除、持久化结账记录、JWT、统一响应和 API 文档等。

## 4. 减少或弱化的功能

### 4.1 预定时间段库存校验未实现

初始需求要求根据房型和预定时间段查询可用数量，无可用房型时拒绝预定。当前 `sp_create_reservation` 只进行以下校验：

- 操作员是否已启用；
- 开始时间是否早于结束时间；
- 入住人数是否大于 0；
- 预付款是否非负。

随后直接写入 `Reservation`，没有统计同房型重叠预定和房间总数。因此多个客户可以在相同时间段预定超过实际数量的同一房型。

**可能原因（工程推断）**：当前预定只关联房型、不提前绑定具体房间。要正确判断库存，需要实现时间区间重叠查询、扣除停用房间、处理并发预定并加锁，复杂度明显高于当前基于即时房态的实现。

**影响**：这是当前与初始业务需求差异最大的一项，建议作为最高优先级补齐。

### 4.2 上级人员审核被收缩为管理员审核

初始需求允许“管理员或对应上级”审核账号。当前：

- 注册模型和数据库仍保留 `SupervisorId`；
- 前端注册固定提交 `supervisorId = null`；
- 审核接口只允许管理员角色调用。

**可能原因（工程推断）**：管理员单点审核更容易实现和验证，可以避免递归上级关系、越级审批和审批权限边界问题。

### 4.3 审核意见没有形成业务记录

前端会发送 `reviewComment`，请求模型也接受该字段，但业务服务只使用审批状态，数据库没有独立的审批记录表或审核意见字段。

**可能原因（工程推断）**：当前以账号状态变更触发的操作日志替代了完整审批流，满足了“谁的状态发生变化”的基础审计，但没有保留审核理由。

### 4.4 多项后端管理能力没有前端入口

当前存在“API 已实现、页面未实现”的功能：

- 修改账号角色、电话、上级人员；
- 查看客户历史业务；
- 修改客户资料和折扣等级；
- 换房；
- 修改、删除房型；
- 完整修改、删除房间；
- 管理员分配保洁员；
- 按多个业务条件查询入住、结账和审计记录。

**可能原因（工程推断）**：项目采用前后端协作开发，后端优先覆盖接口规范，前端优先完成可演示的核心流程，管理型表单和详情弹窗被延后。

### 4.5 “已预定”房态没有自动业务闭环

数据库允许房间状态为“已预定”，但预定只关联房型，不关联具体房间，创建预定时也不会自动把某个房间改为已预定。实际业务主要使用空闲、已入住、待清扫和停用。

**可能原因（工程推断）**：按房型预定符合多数酒店“不提前锁定具体房号”的做法，但初始需求同时保留具体房间“已预定”状态，两个模型没有完全统一。

### 4.6 数据备份未实现

安全需求 S08 要求定期备份核心数据。当前仓库没有：

- 自动备份脚本；
- 定时任务配置；
- 备份恢复说明；
- 备份文件校验或保留策略。

**可能原因（工程推断）**：备份属于部署运维能力，课程项目当前重点在数据库结构和业务闭环，尚未进入生产部署阶段。

### 4.7 异常访问审计不完整

当前会拒绝无效 token、未启用账号和越权访问，但不会把以下行为写入操作日志：

- 登录失败；
- 无效 token；
- 角色越权请求；
- 非法状态修改尝试。

这意味着 S10 只实现了“拒绝处理”，没有完整实现“写入操作记录”。

### 4.8 敏感信息保护只完成了密码部分

账号密码使用 bcrypt 哈希存储，满足“不明文保存”。但客户身份证号和联系电话仍会完整返回并显示，没有字段脱敏、访问目的记录或数据加密。

**可能原因（工程推断）**：当前角色权限被视为主要保护边界，字段级脱敏和加密会增加前端展示、查询和密钥管理复杂度。

## 5. 新增功能

### 5.1 新增审计员角色

初始用户角色表只有管理员、前台工作人员、保洁员和顾客。当前系统增加“审计员”后台角色，权限包括：

- 只读查询客户；
- 查询房型、房态、入住和结账记录；
- 查询操作类型和操作日志；
- 不能修改客户、账号、房间或订单。

**增加原因（工程推断）**：把审计查询从管理员职责中拆分出来，符合最小权限原则，也便于展示角色权限隔离。

### 5.2 新增运营首页总览

前端新增首页仪表盘，展示：

- 客户总数；
- 空闲客房数；
- 待入住预定数；
- 待处理清扫数；
- 房态看板；
- 结账实收汇总；
- 当前在住订单。

**增加原因（工程推断）**：初始需求以业务处理为主，没有综合运营视图。仪表盘提高了系统可观察性和课程演示效果。

### 5.3 新增预定取消

当前增加 `PATCH /reservations/{reservationId}/cancel`，未入住预定可以变更为已取消。

**增加原因**：初始数据字典已经定义“已取消”状态，但功能表没有给出取消操作。当前实现补齐了状态的实际入口。

### 5.4 新增错误入住订单删除

当前增加 `DELETE /check-ins/{checkInId}`，允许管理员或前台删除未结账且没有结账记录的错误入住订单，并恢复房间和关联预定状态。

**增加原因（工程推断）**：用于修正误操作和联调测试产生的错误数据。它属于运维纠错功能，不是正常退房流程的替代品。

### 5.5 新增清扫任务分配和领取机制

初始需求主要描述保洁员查看并完成任务。当前增加：

- 管理员分配或取消分配保洁员的接口；
- 保洁员查看自己的任务和未分配任务；
- 保洁员开始任务时领取任务；
- “待清扫 → 清扫中 → 已完成”完整状态流转。

**增加原因**：让清扫队列从简单完成反馈变为可执行的任务分派流程。

### 5.6 新增独立结账表和账单快照

初始需求定义了结账账单数据结构，但数据存储主要列出 D1-D11，没有明确独立结账表。当前增加 `Checkout` 表，持久化：

- 原价；
- 折扣率快照；
- 折后金额；
- 预付款；
- 实际支付或退款金额；
- 收银员和结账时间。

**增加原因**：即使后续房价或客户折扣发生变化，历史账单仍可保持结账时的金额证据，增强财务可追溯性。

### 5.7 明确支持退款金额

初始完整性需求 I15 写“实际应支付金额不得为负数”，但同一文档的数据字典又说明实际应支付金额可以为负，表示退款。当前实现选择后者：预付款超过折后金额时，实际金额可以为负。

**增加/调整原因**：这是对初始需求内部矛盾的业务澄清，保留退款语义更符合预付款结算场景。

### 5.8 新增 API 工程化能力

当前实现增加了初始业务需求未详细描述的工程能力：

- FastAPI 自动生成 Swagger/OpenAPI 文档；
- JWT Bearer Token 鉴权；
- 统一 `{code, message, data, requestId}` 响应；
- 统一参数校验和数据库错误映射；
- `requestId` 请求追踪；
- `/health` 服务和数据库健康检查；
- CORS 配置；
- 前后端同源部署；
- 本地 `.env` 密钥隔离；
- 开发数据库自动初始化脚本。

**增加原因**：这些能力主要服务前后端协作、问题定位、本地搭建和接口一致性，是将数据库课程设计落地为可运行 Web 系统所必需的技术补充。

### 5.9 新增数据库工程结构

除基础表外，当前数据库还提供：

- `v_available_room`、`v_room_status_overview`、`v_reservation_detail`、`v_checkin_detail`、`v_checkout_bill`、`v_cleaning_task_queue`、`v_operation_audit` 等视图；
- 预定、入住、结账、换房、清扫和审计存储过程；
- 入住、结账和账号状态触发器；
- 面向客户、订单、房态、审计和清扫队列的索引。

**增加原因**：把一致性规则下沉到数据库，减少前后端绕过业务规则直接改表造成的数据不一致。

## 6. 完整性与安全性需求差异

### 6.1 完整性需求

| 范围 | 结论 |
| --- | --- |
| 主键、唯一键和外键 | 基本完整实现，身份证号、账号名、房间号等由数据库唯一约束保证 |
| 状态取值 | 使用 CHECK 约束和业务层校验限制房态、订单、预定、账号和清扫状态 |
| 入住/结账/清扫一致性 | 通过存储过程、事务和触发器实现，整体强于只靠应用层校验 |
| 同一房间重复入住 | 当前房态锁定和事务可防止正常接口下重复入住，但没有按历史时间区间进行通用冲突计算 |
| 预定库存一致性 | 未完成按房型和时间段的容量校验，不能防止预定超卖 |
| 金额约束 | 房价、预付和折扣有约束；实际结算允许负数，以支持退款 |

### 6.2 安全性需求

| 编号 | 当前状态 | 说明 |
| --- | --- | --- |
| S01 登录认证 | 完整 | 账号密码验证、JWT、受保护接口鉴权 |

| S03-S06 角色隔离 | 基本完整 | 后端路由执行角色校验，前端同时隐藏无权限菜单；后端是最终安全边界 |
| S07 操作日志 | 部分 | 主要成功操作有日志，但账号资料修改和异常访问日志不完整 |

| S09 敏感信息保护 | 部分 | 密码已哈希；身份证号和电话未脱敏或加密 |

## 7. 变化原因归纳

### 7.1 为降低课程项目实现复杂度而收缩

- 使用统一登录代替管理员/工作人员两套登录流程；
- 使用管理员单点审批代替上级审批链；
- 前端优先覆盖演示主流程，复杂编辑弹窗和组合查询暂缓；
- 未实现需要并发库存算法的时间段预定容量控制；
- 未建设偏生产运维的数据备份、字段加密和异常安全审计。

### 7.2 为保证数据一致性而增强

- 用存储过程封装预定、入住、结账、换房和清扫；
- 用触发器联动房态、订单和清扫任务；
- 用独立结账表保留账单快照；
- 用唯一约束限制一个房间同时存在多个未完成清扫任务；
- 用事务和行锁处理入住、换房和清扫状态切换。

### 7.3 为前后端协作和演示而增强

- 增加真实 `/api/v1` 接口前端和角色菜单；
- 增加 JWT、统一响应、请求追踪和 API 文档；
- 增加首页运营统计和房态看板；
- 增加 `.env` 配置和本地数据库初始化脚本；
- 增加预定取消和错误数据纠正能力。

## 9. 最终评价

当前系统不是对初始需求的大幅删减，而是形成了一个“核心交易流程完整、管理页面部分简化、工程能力明显增强”的实现版本。

完成度最高的是入住、结账、房态和清扫的一致性闭环；新增价值最高的是持久化结账账单、审计员角色、真实 API 前端和数据库事务化实现。当前最明显的短板是预定时间段库存控制、部分后端能力缺少前端入口，以及备份和异常访问审计未落地。

因此，后续如果以“业务正确性”为目标，应优先补预定库存；如果以“功能展示完整度”为目标，应优先补管理型前端；如果以“可部署安全性”为目标，应优先补备份、敏感信息保护和异常审计。
